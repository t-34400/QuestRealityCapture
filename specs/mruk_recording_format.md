# mruk_recording_format.md

## Purpose

This specification defines the QuestRealityCapture MRUK camera backend recording format.

MRUK output is a new downstream-facing format. It must not be represented as legacy Camera2 raw YUV output unless the recorded bytes and metadata truly match the legacy YUV specification.

---

# Authority

This document is authoritative for MRUK camera frame persistence, MRUK camera metadata, MRUK timestamp and pose metadata, and MRUK stereo pairing.

`recording_session.md` defines backend selection and session orchestration.

`legacy_recording_format.md` defines the legacy Camera2/NativeCamera2 recording layout.

`yuv_storage_format.md` defines compatibility-sensitive YUV persistence requirements and does not apply to MRUK RGBA frame files.

---

# Compatibility Boundary

The MRUK backend must be treated as a distinct recording backend from Camera2/NativeCamera2.

MRUK frames must not be written into the legacy camera directories unless the implementation writes actual legacy-compatible Camera2 `YUV_420_888` plane buffers and matching format metadata.

The following legacy paths are reserved for Camera2-compatible YUV output:

```text
left_camera_raw/
right_camera_raw/
left_camera_image_format.json
right_camera_image_format.json
left_camera_characteristics.json
right_camera_characteristics.json
stereo_pairs.csv
```

The MRUK backend must use MRUK-specific paths so downstream tools cannot accidentally parse RGBA frames as raw YUV frames.

---

# Session Layout

An MRUK camera session should write camera-specific output using the following layout:

```text
<session>/
    session_info.json

    left_camera_mruk_rgba/
    right_camera_mruk_rgba/

    left_camera_mruk_frame_metadata.csv
    right_camera_mruk_frame_metadata.csv

    left_camera_mruk_intrinsics.json
    right_camera_mruk_intrinsics.json

    mruk_stereo_pairs.csv
```

Depth and pose outputs outside the camera backend may continue to use their existing session paths unless a separate specification changes them.

---

# Frame Files

MRUK frame files should store the actual Unity-side pixel buffer obtained from MRUK without pretending to be Camera2 YUV output.

The initial MRUK frame format is:

```text
RGBA32 byte data
```

Recommended file extension:

```text
.rgba
```

Frame file names should use the MRUK frame Unix timestamp in microseconds when available:

```text
<timestamp_us_realtime>.rgba
```

If the implementation cannot obtain a stable microsecond timestamp for filenames, it must record the exact naming basis in the per-frame metadata CSV and in `session_info.json` or a backend metadata file.

---

# Per-Frame Metadata

Each MRUK camera side must write a per-frame metadata CSV.

The CSV should include at least:

```csv
frame_index,file_name,unix_time_ms,timestamp_us_realtime,width,height,format,camera_position,pose_pos_x,pose_pos_y,pose_pos_z,pose_rot_x,pose_rot_y,pose_rot_z,pose_rot_w
```

`timestamp_us_realtime` is the Unix-epoch microsecond timestamp represented by `PassthroughCameraAccess.Timestamp` when available.

`pose_*` values must come from MRUK camera pose for the recorded frame, not from a later current-frame HMD pose sample.

The preferred pose source is:

```text
PassthroughCameraAccess.GetCameraPose()
```

The metadata format may add fields for monotonic/XrTime-based timestamps when they become available through a stable integration path. Added fields must not change the meaning of existing fields.

---

# Intrinsics Metadata

Each MRUK camera side must write MRUK intrinsics metadata separately from legacy Camera2 characteristics metadata.

Recommended file names:

```text
left_camera_mruk_intrinsics.json
right_camera_mruk_intrinsics.json
```

The JSON should include at least:

```json
{
  "backend": "MRUK",
  "cameraPosition": "Left",
  "imageFormat": "RGBA32",
  "focalLength": {
    "x": 0.0,
    "y": 0.0
  },
  "principalPoint": {
    "x": 0.0,
    "y": 0.0
  },
  "resolution": {
    "width": 0,
    "height": 0
  },
  "lensOffset": {
    "position": {
      "x": 0.0,
      "y": 0.0,
      "z": 0.0
    },
    "rotation": {
      "x": 0.0,
      "y": 0.0,
      "z": 0.0,
      "w": 1.0
    }
  }
}
```

The stored values must describe the actual MRUK frame dimensions and camera model used for the recorded frames.

# Stereo Pairing

When both MRUK camera sides are recorded, pairing metadata should be written at the session root:

```text
mruk_stereo_pairs.csv
```

The CSV should include at least:

```csv
pair_index,left_file_name,right_file_name,left_timestamp_us_realtime,right_timestamp_us_realtime,time_delta_us
```

Pairing metadata must be separate from frame files.

Pair rows must be generated by timestamp proximity over recorded left/right frame metadata, not by assuming that equal frame indices or directory order imply a stereo pair. The recommended pairing rule is to match each recorded frame with at most one opposite-side frame while minimizing absolute `time_delta_us` among unpaired candidates.

Consumers that require stereo associations should use `mruk_stereo_pairs.csv` instead of inferring pairs from directory order.

---

# Timestamp and Pose Semantics

MRUK sessions must preserve the distinction between public Unix-epoch timestamps and runtime pose lookup timestamps.

The public frame timestamp is represented by MRUK as a Unix-epoch `DateTime` and should be recorded as Unix time in milliseconds and microseconds when available.

Frame pose should be derived from MRUK's frame-timestamped camera pose API rather than from the current HMD pose logger.

The preferred MRUK pose API is:

```text
PassthroughCameraAccess.GetCameraPose()
```

This is the main reason to prefer MRUK for synchronized camera recording.

---

# Validation Requirements

Changes affecting MRUK persistence must validate that:

* MRUK frame files are not written to legacy YUV directories.
* MRUK frame metadata describes the actual saved byte format.
* Intrinsics metadata matches the recorded frame resolution.
* Per-frame pose metadata is sampled from MRUK camera pose for the recorded frame.
* Stereo pair metadata references existing left and right frame files.

Prefer targeted validation over broad test execution.
